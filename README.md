# deepseek-pg-perf-prompts
Коллекция промптов для нейросети **DeepSeek** (на русском языке), предназначенных для анализа и оптимизации производительности СУБД **PostgreSQL**.


---

## 📦 Требования

- PostgreSQL 15+ (проверялось на версиях 15 и выше)
- Аккаунт DeepSeek (бесплатного тарифа достаточно для большинства промптов)
- Базовое знание SQL для подстановки своих значений

## 📘 Инструкция для анализа производительности

В репозитории присутствует файл **`_prompt-instruction.txt`** – это детальная инструкция (промпт) для нейросети DeepSeek, задающая роль эксперта по производительности PostgreSQL и строгие правила анализа.

### Что содержит инструкция?

- Роль: эксперт по производительности СУБД PostgreSQL.
- Правила ответов: опора только на предоставленные данные, указание недостающих метрик, профессиональная терминология (shared_buffers, seq scan, autovacuum и т.д.).
- Уровни достоверности выводов (от «подтверждено данными» до «невозможно оценить»).
- Протокол обработки противоречий в данных.
- Требования к стилю и границы применимости.

# Сравнительный анализ СУБД по отчетам pgpro_pwr

## 1. `parameter table`

**Назначение**: Формирование сравнительной таблицы конфигурационных параметров кластера PostgreSQL на основе двух отчётов `pgpro_pwr`.

**Файлы директории**:
- `readme.md` — инструкция по трёхэтапному процессу сравнения параметров.
- `prompt-correspondence_table.source.1.txt` — шаблон запроса для извлечения параметров из первого отчёта.
- `prompt-correspondence_table.source.2.txt` — шаблон запроса для извлечения параметров из второго отчёта.
- `prompt-correspondence_table.result.txt` — шаблон запроса для формирования итоговой сравнительной таблицы.
- `Таблица.G.78.txt` — описание раздела «Cluster settings during the report interval».

**Краткое описание этапов**:
1. **Этап 1**: Извлечение параметров из первого отчёта (`pgpro_pwr.X1-X2.clear.html`). Загружаются `_prompt-instruction.txt`, HTML-отчёт и `Таблица.G.78.txt`. После замены плейсхолдера в `prompt-correspondence_table.source.1.txt` результат сохраняется как `table.source.1.txt`.
2. **Этап 2**: Извлечение параметров из второго отчёта (`pgpro_pwr.Y1-Y2.clear.html`). Аналогично этапу 1, результат → `table.source.2.txt`.
3. **Этап 3**: Формирование итоговой таблицы. Загружаются `_prompt-instruction.txt`, `table.source.1.txt`, `table.source.2.txt` и исправленный `prompt-correspondence_table.result.txt`. Результат → `table.result.txt`.

**Особенности**:
- Итоговая таблица `table.result.txt` содержит только строки параметров (`Setting`), значения которых различаются или отсутствуют в одном из отчётов.
- Формат: `Setting | отчет-X1-X2 | отчет-Y1-Y2`.

## 2. `Load_distribution`

**Назначение**: Извлечение таблицы «Load distribution» из HTML-отчёта `pgpro_pwr` с помощью специального промпта.

**Файлы директории**:
- `readme.md` — инструкция по получению таблицы распределения нагрузки.
- `prompt-load.source.txt` — техническое задание для извлечения данных из `pgpro_pwr.X1-X2.clear.html`. Содержит правила фильтрации (только строки с `db_filter: true`), маппинг метрик из `ТаблицаG.25.txt` на поля JSON и требования к форматированию итоговой markdown-таблицы.
- `ТаблицаG.25.txt` — словарь метрик с пояснениями и формулами расчёта (Total time, Executed count, I/O time, Blocks fetched, Shared blocks read/dirtied/written, WAL generated и др.).

**Взаимосвязь файлов**:
- `ТаблицаG.25.txt` задаёт словарь метрик.
- `prompt-load.source.txt` использует этот словарь для маппинга на поля JSON и определяет структуру выходной таблицы.

**Краткое описание процесса**:
1. В новый чат загружается `_prompt-instruction.txt` (из корня репозитория) и вводится фраза «Строго следуй инструкции, ожидай данные».
2. Редактируется `prompt-load.source.txt`: плейсхолдер `X1-X2` заменяется на фактические значения из имени отчёта.
3. Загружаются HTML-отчёт и `ТаблицаG.25.txt`.
4. Отправляется исправленный промпт; результат сохраняется как `load_distribution.1.txt` (или `.2.txt` для второго отчёта).

## 3. `Load distribution compare`

**Назначение**: Сравнительный анализ таблиц распределения нагрузки (`Load distribution`) из двух отчётов `pgpro_pwr` с учётом изменений конфигурационных параметров СУБД.

**Файлы директории**:
- `readme.md` — последовательность действий для подготовки аналитического отчёта.
- `_prompt-instruction.txt` — базовая инструкция для ИИ-эксперта (системный промпт).
- `prompt-analyze.txt` — шаблон аналитического запроса с плейсхолдерами `X1-X2`, `Y1-Y2`, которые заменяются на реальные идентификаторы отчётов.
- `Таблица_G.78.txt` — описание раздела «Cluster settings during the report interval» (словарь параметров кластера).
- `ТаблицаG.25.txt` — описание метрик распределения нагрузки (`Total time`, `Executed count`, `I/O time` и др.).

**Краткое описание процесса**:
1. Создаётся новый чат, загружаются `_prompt-instruction.txt` и `Philosophical_instruction_BETA_v5.1.md` (из корня репозитория).
2. Последовательно отправляются файлы: `load_distribution.1.txt`, `load_distribution.2.txt`, `Таблица_G.78.txt`, `ТаблицаG.25.txt`, `table.result.txt`.
3. Редактируется и отправляется `prompt-analyze.txt` с подставленными идентификаторами отчётов.
4. Результат сохраняется как `analyze.txt`.

## 4. `Top-SQL analyze`

**Назначение**: Комплексный пятиэтапный анализ, объединяющий сравнение конфигурационных параметров, распределения нагрузки, сопоставление SQL-запросов по тексту, извлечение и сравнение планов выполнения, а также формирование итогового аналитического отчёта.

**Файлы директории**:
- `readme.md` — пошаговое руководство по пяти этапам.
- `prompt-1.txt` — запрос для извлечения списка Top SQL из второго отчёта (Этап 1).
- `prompt-2.txt` — запрос для сопоставления Query ID между отчётами по тексту SQL (Этап 2).
- `prompt-3.txt` — запрос для извлечения планов выполнения из второго отчёта (Этап 3).
- `prompt-4.txt` — запрос для извлечения планов выполнения из первого отчёта (Этап 4).
- `prompt-analyze.txt` — шаблон итогового аналитического запроса с анализом планов (Этап 5).

**Краткое описание этапов**:
1. **Этап 1**: Извлечение списка SQL-запросов из второго отчёта → результат `queryid-1.txt`.
2. **Этап 2**: Сопоставление Query ID между отчётами по тексту SQL → `queryid-2.txt`.
3. **Этап 3**: Извлечение планов выполнения из второго отчёта (используется `queryid-2.txt`) → `plan-3.txt`.
4. **Этап 4**: Извлечение планов выполнения из первого отчёта (используется `queryid-2.txt`) → `plan-4.txt`.
5. **Этап 5**: Формирование итогового аналитического отчёта. Загружаются `table.result.txt`, `analyze.txt`, `plan-3.txt`, `plan-4.txt`, `queryid-2.txt`, `Таблица_G.78.txt`, `ТаблицаG.25.txt`, а также инструкции `_prompt-instruction.txt` и `Philosophical_instruction_BETA_v5.1.md`. Результат → `analyze-Top_SQL.txt`.

**Примечания**:
- На всех этапах обязательна загрузка `_prompt-instruction.txt` и фраза «Строго следуй инструкции, ожидай данные».
- Цветовые маркеры из `Philosophical_instruction_BETA_v5.1.md` заменяются на текстовые метки: «Подтверждено», «Вероятно», «Предположение», «Неизвестно».
